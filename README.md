# Приложение для управления задачами

Это приложение является **системой управления задачами**, которая поддерживает **трехзвенную архитектуру** и использует различные технологии для взаимодействия между клиентом и сервером, а также аутентификации пользователей и ролевой модели доступа.

## 1. **Обзор архитектуры**

Приложение использует **трехзвенную архитектуру**:
- **Клиентская часть**: фронтенд-приложение (создано с использованием React)
- **Сервер приложений**: сервер (создан с использованием FastAPI)
- **Сервер базы данных**: PostgreSQL (контейнеризован с использованием Docker)

### 1.1 **Протокол взаимодействия**:
- Взаимодействие между клиентом и сервером осуществляется через **HTTP** с использованием **RESTful API**.
- Выбран **HTTP**, так как это широко используемый и поддерживаемый протокол для веб-приложений, и он хорошо работает с **JSON** для обмена данными.

### 1.2 **Клиентское приложение**:
- Клиент — это **веб-приложение**, созданное с использованием **React**. Оно взаимодействует с сервером через HTTP-запросы.
- Клиент предоставляет пользовательский интерфейс для:
  - Просмотра, создания, обновления и удаления задач
  - Обработки аутентификации и авторизации пользователей
  - Администрирования пользователей (для пользователей с ролью администратор)

### 1.3 **Сервер приложений**:
- Сервер построен с использованием **FastAPI**, современного, быстрого и асинхронного веб-фреймворка для создания API на Python 3.7+.
- Сервер обрабатывает:
  - Аутентификацию пользователей
  - Управление задачами
  - Ролевую модель доступа
  - Взаимодействие с базой данных

### 1.4 **База данных**:
- В качестве базы данных используется **PostgreSQL**, которая хранит:
  - Данные пользователей (имя пользователя, почта, хэш пароля, роль)
  - Данные задач (заголовок, описание, статус выполнения, связанный пользователь)

### 1.5 **Сетевое взаимодействие**:
- Взаимодействие между клиентом и сервером происходит через **HTTP** запросы.

## 2. **Дизайн API**

API следует принципам REST. Все взаимодействие клиента и сервера происходит через HTTP-запросы. Ниже приведены основные операции API:

### 2.1 **Аутентификация**:
- **POST /login**:
  - **Запрос**: { "username": "user1", "password": "password123" }
  - **Ответ**: { "access_token": "jwt_token", "token_type": "bearer" }
  
- **POST /register**:
  - **Запрос**: { "username": "user1", "password": "password123", "email": "user1@example.com", "role": "editor" }
  - **Ответ**: { "access_token": "jwt_token", "token_type": "bearer" }

### 2.2 **Управление задачами**:
- **GET /tasks**:
  - **Запрос**: Без тела запроса
  - **Ответ**: Список задач для авторизованного пользователя (если роль **editor** или **moderator**). **Moderator** может видеть все задачи.
  - **Ошибка**: 401 Unauthorized (если токен отсутствует или неверен).

**GET /tasks/{task_id}**:
  - **Запрос**: { "task_id": "1" }
  - **Ответ**: Получение данных задачи по её **ID**.
  - **Ошибка**:
    - **404 Not Found**: Задача не найдена или пользователь не авторизован для её просмотра.
    - **403 Forbidden**: Пользователь не имеет прав для просмотра задачи.
    - **401 Unauthorized**: Отсутствует или неверен токен.
  
- **POST /tasks**:
  - **Запрос**: { "title": "New Task", "description": "Task description", "is_completed": false }
  - **Ответ**: Созданная задача с её деталями.
  
- **PUT /tasks/{task_id}**:
  - **Запрос**: { "title": "Updated Task", "description": "Updated description", "is_completed": true }
  - **Ответ**: Обновленная задача.
  
- **DELETE /tasks/{task_id}**:
  - **Ответ**: { "message": "Task deleted successfully" }

### 2.3 **Управление пользователями** (только для **admin**):
- **GET /users**:
  - **Запрос**: Без тела запроса
  - **Ответ**: Список всех пользователей (кроме самого **admin**).
  
- **DELETE /users/{user_id}**:
  - **Запрос**: Без тела запроса
  - **Ответ**: { "message": "User deleted successfully" }

### 2.4 **Коды ошибок**:
- **400 Bad Request**: Неверные данные в запросе.
- **401 Unauthorized**: Отсутствует или неверен токен.
- **403 Forbidden**: У пользователя нет прав для выполнения действия.
- **404 Not Found**: Ресурс не найден (например, задача или пользователь).
- **422 Unprocessable Entity**: Ошибка валидации данных.

## 3. **База данных**

Приложение использует **PostgreSQL** для хранения данных. База данных состоит из двух основных таблиц:
- **Таблица пользователей**:
  - **id** (Primary Key)
  - **username** (Unique)
  - **email** (Unique)
  - **hashed_password**
  - **role** (может быть "editor", "moderator", "admin")

- **Таблица задач**:
  - **id** (Primary Key)
  - **title**
  - **description**
  - **is_completed**
  - **owner_id** (Foreign Key, ссылается на таблицу Users)

### 3.1 **Настройка базы данных**:
База данных контейнеризирована с использованием **Docker**. Для оркестрации контейнеров используется **`docker-compose.yml`**.

## 4. **Аутентификация и авторизация пользователей**

- Приложение использует **JWT (JSON Web Token)** для аутентификации пользователей.
- Реализованы следующие роли:
  - **admin**: Полный доступ ко всем задачам и пользователям (может изменять задачи любых пользователей, удалять пользователей и изменять роли).
  - **moderator**: Может просматривать и изменять задачи любого пользователя, но не может управлять пользователями.
  - **editor**: Может просматривать и изменять только свои задачи.

### 4.1 **Генерация токенов**:
- Во время логина или регистрации генерируется **JWT токен**, который используется для аутентификации пользователя в последующих запросах.

### 4.2 **Контроль доступа по ролям**:
- **admin**: Полный доступ ко всем задачам и пользователям.
- **moderator**: Может только просматривать и редактировать задачи, не имея прав на управление пользователями.
- **editor**: Может просматривать и редактировать только свои задачи.

## 5. **Используемые технологии**

- **Frontend**:
  - **React**: Для создания динамичного и отзывчивого пользовательского интерфейса.
  - **React Router**: Для маршрутизации на клиенте.
  - **Axios**: Для выполнения HTTP-запросов на сервер.

- **Backend**:
  - **FastAPI**: Современный веб-фреймворк для создания API на Python.
  - **SQLAlchemy**: ORM для взаимодействия с PostgreSQL.
  - **JWT (JSON Web Token)**: Для аутентификации.

- **Database**:
  - **PostgreSQL**: Реляционная база данных для хранения данных (задачи, пользователи, роли).

## 6. **Запуск приложения**

### 6.1 **Предварительные требования**
- Установлен **Docker** для контейнеризации.

### 6.2 **Инструкции по настройке**

#### Backend (FastAPI & PostgreSQL):
1. Соберите и запустите frontend, backend и базу данных с помощью **Docker Compose**:

```bash
docker-compose up --build
```
Доступ к серверу backend через http://localhost:8000

Swagger доступен по http://localhost:8000/docs

Доступ к фронтенду через http://localhost:3000

### 6.3 API Эндпойнты

- Login: POST /login

- Register: POST /register

- Tasks:

  - GET /tasks

  - POST /tasks

  - PUT /tasks/{task_id}

  - DELETE /tasks/{task_id}
  
- Users (Admin Only):
  
  - GET /users
  
  - PUT /users/{user_id}
  
  - DELETE /users/{user_id}
